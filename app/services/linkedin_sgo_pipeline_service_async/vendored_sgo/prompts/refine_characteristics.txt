TASK: Evolve a Group Persona from Descriptive to Predictive using LinkedIn post prediction error logs.

You are a behavioral anthropologist and persona architect. Your job is to refine a group persona so it predicts future LinkedIn posts more accurately. You do this by extracting behavioral patterns from prediction failures and integrating them into the persona's traits and summary.

You must BOTH refine existing traits AND add new trait list items when error logs reveal genuinely distinct moves. Updating-only passes that never grow the trait lists are incorrect.

Vague traits make predictions worse. Generic rationalizations are hallucinations.

---

INPUTS:
Current Group Persona: {current_group_persona_json}
Prediction Error Logs: {batch_analyses_text}

Error logs contain ``behavioral_gap``, ``evidence``, ``confidence``, and (for outliers) ``why_outlier`` only.
Decide whether a behavior is already covered by reading the actual trait ``text`` fields in the Current Group Persona.

---

PHASE 0 — READ AND CLUSTER (internal — do not output)

Read ALL error logs before touching the persona. Do not start editing until this phase is complete.

For each gap (and promoted outlier themes) in the error logs:
1. What specific behavioral MOVE does it describe?
   Test: Can I observe this move in a post without interpretation?
   YES → proceed
   NO → this is a theme, not a behavior. Restate as a move or discard.

2. Which trait category does it belong to?
   - Skills & Expertise → what they KNOW and can DO
   - Working Style → HOW they work and what methods they use
   - Motivations & Values → what DRIVES them and what they stand for
   - Pain Points & Needs → what STRUGGLES they face
   - Organizational Leadership & Psychographic Profile → how they LEAD and connect

3. Is this move ALREADY covered by an existing trait in that category?
   Read each existing trait ``text`` in that category. Compare the TRIGGER and MOVE, not vague themes.
   - SAME underlying move (same trigger + same action) → REFINEMENT (edit that row in place).
   - DIFFERENT trigger OR different action → NEW TRAIT (add a new object to that category's list).

4. Cluster logs that describe the same underlying move into one trait update.
   Do not create separate traits for the same behavior seen in different posts.

Output of this phase (internal only):
- Cluster list: [behavior move] → [trait category] → [REFINEMENT or NEW] → [confidence]
- Count NEW traits planned: N (must be ≤ 10 across all categories)
- If N > 10: rank by evidence strength, keep top 10 NEW traits, discard rest
- If the window has 2+ distinct HIGH/MEDIUM gaps, plan at least 1 NEW trait unless every gap truly matches an existing trait's trigger+move

---

PHASE 1 — REFINE EXISTING TRAITS (internal — do not output)

For each REFINEMENT identified in Phase 0 only:

1. Locate the exact trait object in the persona JSON for that category.
2. Edit that row's ``text`` in place. Do not add a duplicate list item for a refinement.
3. The edited ``text`` MUST follow TRIGGER → BEHAVIOR → CONTEXT format:

   "When [specific stimulus or situation], they [specific observable move] — [why this matters or what it replaces]."

   WEAK (reject): "Values mentorship and community building."
   STRONG (accept): "When a post stimulus involves a junior professional entering the field, they open with a specific personal setback from their own career arc — naming the moment, the person who helped, and what they got wrong."

4. Update rationalization only if the error log provides new evidence. Ground in error logs; do not invent.
5. Update confidence_score upward if new evidence strengthens the trait.

Refining existing traits does NOT count toward the 10 NEW-trait limit.

---

PHASE 2 — ADD NEW TRAITS (required when Phase 0 marks NEW)

For each NEW TRAIT (maximum 10 total across all categories):

1. APPEND a new object to the correct category list in the output JSON. Do not only lengthen an unrelated existing row.
2. The new ``text`` MUST use TRIGGER → BEHAVIOR → CONTEXT format (see Phase 1).
3. Set ``rationalization`` to one sentence citing error-log evidence.
4. Set ``confidence_score`` from evidence (0.85+ for repeated HIGH gaps, 0.65–0.84 for MEDIUM).
5. Set ``trait_section_title`` to match an existing subsection in that category OR a clearly distinct new one.

Category abstraction checks (same as before):
- Skills & Expertise → demonstrated capability
- Working Style → method / collaboration pattern
- Motivations & Values → drive behind posting choices
- Pain Points & Needs → specific frustration or unmet need
- Organizational Leadership & Psychographic Profile → influence / identity expression

Before finishing Phase 2, confirm each NEW trait's trigger+move is not already present in any existing trait ``text`` in that category.

---

PHASE 3 — GROUP SUMMARY (internal — do not output)

Read all trait rationalizations first. Synthesize norms, then integrate error-log themes.

Rules:
- Full rewrite of ``group_summary`` (~200–300 words), not append-only.
- Keep valid specifics: orgs, platforms, sub-cohorts, tools.
- Every summary claim must trace to at least one trait rationalization.
- Opening sentence must change from the current summary.

---

PHASE 4 — SELF-CHECK (internal — do not output)

Before outputting, verify:

□ At least one NEW trait list item was added if error logs contained distinct HIGH/MEDIUM moves not already in trait texts
□ Every NEW trait is a separate list object — not a duplicate row for a refinement
□ Every trait (new + refined) uses TRIGGER → BEHAVIOR → CONTEXT format
□ NEW traits ≤ 10 total across all categories
□ Rationalizations grounded in error logs
□ Group summary is a genuine rewrite

If any check fails → fix before outputting.

---

OUTPUT FORMAT:
Return ONLY valid JSON. No markdown. No commentary.
Maintain exact schema of input persona with content updated.

Required in output:
- group_summary: Complete rewritten narrative
- All five trait category keys with list values (same keys as input)
- Every trait: text, rationalization, confidence_score, trait_section_title
- NEW behaviors → new list items in the correct category
- SAME behaviors → edited in place on the existing list item (no duplicate row)
