1. Read all logs; find recurring patterns (e.g. “celebration / visibility” vs “deep technical how-to”).
2. For each pattern, ask what **trait**, **motivation**, or **goal** was missing for better topic predictions.

4. Assess evidence strength (multiple batches vs single post).
5. Plan: refine vs add; which section; merge vs new; **at most 10 NEW object rows** across `inherent_behavioral_traits`, `latent_motivations.main`, and `implicit_goals` combined this step (refinements of existing rows do **not** count toward 10).

### CONFIDENCE SCORING (every object row you add or materially change)
- **0.9–1.0:** Pattern in multiple batches or clearly central in 3+ log lines.
- **0.7–0.8:** Strong in one batch or solid inference from several minor misses.
- **0.5–0.6:** Weak / single-post specific — use sparingly.
Merging: prefer the **higher** confidence among merged items unless evidence clearly warrants a weighted blend.

### SECTIONS (use all three object sections)
- **`inherent_behavioral_traits`:** Observable tendencies in how they **post** or position content.
- **`latent_motivations.main`:** Drives behind those behaviors (recognition, craft pride, risk framing, etc.) **only if grounded in logs**.
- **`implicit_goals`:** Outcomes they seek from posting (hiring funnel, education, advocacy…) **only if grounded in logs**.

**Strict object shape** for every item in those three lists:
```json
{{"text": "[When X in post/context, they …].", "rationalization": "Why: tied to specific log evidence.", "confidence_score": 0.0}}
```
- **Every** object must have `text`, `rationalization`, `confidence_score`.
- **No hallucinated rationalizations:** derive only from log wording/implications.

### SUPPORTING LISTS (strings OK)

### STYLE
- **No shallow labels** (“tech-savvy”). Use **conditional** triggers: when … then …
- Generalize across posts; remove one-off product names unless the log forces them.
- **Evidence-only:** do not invent companies, roles, or motives absent from logs.

### ANTI-OVERFITTING
No laundry list of one log = one trait. Merge overlapping ideas. Describe **persona type**, not one author’s biography.

### NO DUPLICATE TRAITS VS CURRENT PERSONA
Before adding a **new** row, scan the **CURRENT PERSONA** for an existing trait/motivation/goal that already expresses the same trigger or meaning. If one exists, **refine that row’s text** (and rationalization / confidence) instead of appending a near-duplicate. Only use the **≤10 new rows** budget for genuinely **new** behavioral dimensions the current schema does not already encode.
